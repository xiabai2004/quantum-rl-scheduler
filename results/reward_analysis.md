# Reward 设计分解说明

> 对应 Issue #28：完善 `env.py` 奖励函数文档和可视化。
> 目标读者：第一次阅读调度环境的新队友。

## 一句话理解

奖励函数的目标是让智能体学会：

1. 把任务送到兼容的资源上。
2. 在量子资源质量好、可用时优先使用量子加速。
3. 避免队列里任务等待过久。
4. 避免量子比特长期空闲。

当前奖励不是单独一个 `_compute_reward()` 函数，而是由 `step()` 汇总多个部分：

```text
单步总奖励 =
    动作兼容性结果
  + 执行成功奖励
  + 队列等待惩罚
  + 低量子利用率惩罚
```

核心代码位置：

| 位置 | 作用 |
| --- | --- |
| `src/scheduler/env.py::step()` | 汇总单步 reward，处理错误分配、量子不可用、等待惩罚和低利用率惩罚 |
| `src/scheduler/env.py::_compute_execution_reward()` | 计算任务成功执行后的正向奖励 |
| `src/scheduler/env.py::_compute_wait_penalty()` | 计算队列中等待过久任务的惩罚 |

## 奖励常量

| 常量 | 当前值 | 含义 |
| --- | ---: | --- |
| `REWARD_CLASSICAL` | `5.0` | 经典执行基准奖励 |
| `REWARD_QUANTUM_BASE` | `10.0` | 量子执行基础奖励，会乘以量子加速比 |
| `REWARD_HYBRID` | `7.0` | 混合执行基础奖励，介于经典和量子之间 |
| `REWARD_SUCCESS_BONUS` | `3.0` | 任务成功完成额外奖励 |
| `REWARD_MISMATCH` | `-2.0` | 任务分配到不兼容资源的惩罚 |
| `REWARD_WAIT_OVER_THRESHOLD` | `-0.1` | 等待超过阈值后的比例惩罚系数 |
| `REWARD_LOW_QUBIT_UTIL` | `-1.0` | 量子利用率低于 30% 时的惩罚 |
| `QUANTUM_SPEEDUP_RANGE` | `(2.0, 5.0)` | 量子加速比采样范围 |
| `MAX_WAIT_STEPS` | `50` | 超过该等待步数后开始惩罚 |
| `QUBIT_UTIL_THRESHOLD` | `0.3` | 量子利用率低阈值 |

## 执行奖励分解

| 执行动作 | 公式 | 直觉解释 |
| --- | --- | --- |
| 经典执行 | `5.0 + 3.0 = 8.0` | 稳定、可靠，但没有量子加速 |
| 量子执行 | `10.0 * speedup * fidelity_factor + 3.0` | 量子资源质量越高，收益越大 |
| 低保真量子执行 | `(10.0 * speedup * fidelity_factor) * 0.6 + 3.0` | 保真度低于 0.9 时打折，避免盲目用低质量量子资源 |
| 混合执行 | `7.0 * (0.5 + 0.5 * available_ratio) + 3.0` | 量子资源越空闲，混合执行越接近满收益 |

说明：

- `speedup` 从 `2.0` 到 `5.0` 随机采样。
- `fidelity_factor = self._quantum.fidelity / 0.99`。
- `available_ratio` 表示当前量子比特空闲比例，范围是 0 到 1。

## 典型场景表

| 场景 | 输入状态 | 奖励组成 | 预期效果 |
| --- | --- | --- | --- |
| 经典任务走经典 | `task_type=classical`, `action=0` | `+8.0`，再叠加等待/利用率全局项 | 鼓励经典任务走经典资源 |
| 量子任务走量子，保真度高 | `task_type=quantum`, `action=1`, `fidelity≈0.99` | `10 * speedup + 3`，通常大于经典奖励 | 学会在量子资源质量好时使用量子加速 |
| 量子任务走量子，保真度低 | `fidelity<0.9` | 量子执行奖励乘 `0.6` 后再加成功奖励 | 降低低质量量子机器吸引力 |
| universal 任务走混合 | `task_type=universal`, `action=2` | `7 * hybrid_factor + 3` | 在经典和量子之间折中 |
| quantum 任务走经典 | `task_type=quantum`, `action=0` | `-2.0`，任务重新入队 | 惩罚错误路由 |
| classical 任务走量子 | `task_type=classical`, `action=1` | `-2.0`，任务重新入队 | 避免浪费量子资源 |
| 量子资源不可用，纯量子动作 | `action=1`, 无可用量子机器 | `-1.0`，任务重新入队 | 区分资源暂不可用和完全错误分配 |
| 量子资源不可用，混合动作 | `action=2`, 无可用量子机器 | 降级为经典执行，约 `+8.0` | 防止系统空转 |
| 队列任务等待过久 | `wait_steps > 50` | `-0.1 * (wait_steps - 50) / 50` | 鼓励处理快超时任务 |
| 量子利用率过低 | 实际利用率 `< 30%` | `-1.0` | 鼓励合理使用量子资源 |
| 当前没有任务 | `_current_task is None` | `-1.0` | 轻微惩罚空转 |

## 为什么这样设计

### 1. 先保证任务分配正确

兼容性惩罚是第一层约束。任务如果被送到不兼容资源，本步不会执行成功，只会扣分并重新入队。这样智能体会先学会最基本的规则：不要把 quantum-only 任务送到经典资源，也不要把 classical-only 任务送到量子资源。

### 2. 再鼓励高质量量子加速

量子执行的奖励上限高于经典执行，因为项目目标是提高量子资源利用率。但量子奖励会受到保真度影响，保真度低时还会打折。这避免了策略只因为“量子奖励高”就无脑选择量子。

### 3. 混合执行作为折中方案

混合执行奖励介于经典和量子之间，并受量子可用率影响。量子资源越充足，混合执行收益越高；量子资源紧张时，混合执行收益降低。

### 4. 全局惩罚防止局部最优

等待惩罚和低利用率惩罚不是某个任务独有的，而是对整个系统状态的提醒：

- 队列里有人等太久，要扣分。
- 量子资源太闲，也要扣分。

这能让智能体从“只看当前任务”转向“照顾整个调度系统”。

## 新人阅读建议

如果你第一次读奖励逻辑，建议按这个顺序看：

1. 先看 `ACTION_CLASSICAL / ACTION_QUANTUM / ACTION_HYBRID` 三个动作定义。
2. 再看 `REWARD_*` 常量，理解每个分数的意义。
3. 然后看 `step()` 中的兼容性判断和全局奖励累加。
4. 最后看 `_compute_execution_reward()`，理解经典、量子、混合三类执行收益。

读完后可以运行：

```bash
python -m pytest tests/ -v
```

确保奖励相关改动没有破坏现有环境行为。
